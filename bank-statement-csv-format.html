Is your bank statement CSV ready for reconciliation?

Before a CSV export can be matched against expected payments, the risky parts are usually the same: unclear date formats, comma-vs-dot decimals, empty references, duplicate exports, unmapped columns, and descriptions that contain sensitive personal data.

Safe checklist: send facts, not data

For a first fit check, do not send a real bank statement, account number, transaction rows, customer names, references, descriptions, credentials, or personal data. Send only these scope facts:

  1. Column names only, for example Date;Amount;Reference;Description.
  2. Date and decimal convention, for example 31.12.2026 and 123,45.
  3. Approximate monthly row count and number of expected payments.
  4. The desired read-only outcome: missing payments, late payments, duplicate candidates, or import validation warnings.
  5. Confirmation that no bank export, transaction rows, account identifiers, credentials, or personal/customer data are attached.

What the €249 audit checks

  • One bank CSV format and one expected-payment ledger.
  • Column mapping and validation warnings for ignored columns, skipped blank rows, and empty reference/description fields.
  • Matched, missing, and potentially duplicated payments in Markdown and JSON.
  • A local Python runner/config that can be rerun on the same export shape.
